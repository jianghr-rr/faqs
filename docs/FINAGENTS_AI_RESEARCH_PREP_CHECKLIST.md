## 1. 这份清单是干什么的

这份文档不是实现文档，而是开工前的准备清单。

它回答的问题是：

- 在正式写代码前，哪些事情必须先确认
- 哪些内容没有冻结，后面容易反复返工
- 对前端最关键的输入和输出有哪些

建议使用方式：

- 每完成一项，就手动打勾
- 没有确认的项，先不要急着进入开发
- 优先完成 P0，再做 P1

## 2. P0：必须先确认的事情

### 2.1 产品范围

- [ ] 明确第一版是否只做登录用户可用的 AI 投研能力
- [ ] 明确第一版要包含的页面：新闻详情页 / 智能分析页 / 聊天页 / 图谱页
- [ ] 明确第一版哪些页面必须先做，哪些页面可以后置
- [ ] 明确第一版不做复杂图谱可视化，只先做卡片、列表、表格、推理路径展示
- [ ] 明确第一版不是“全市场智能投研”，而是试点行业 MVP

### 2.2 用户与权限

- [ ] 确认 AI 投研相关功能仅登录用户可用
- [ ] 确认未登录用户访问投研 API 时统一返回 `401`
- [ ] 确认未登录用户在页面上看到的是“引导登录”还是“直接跳登录”
- [ ] 确认哪些能力是公开的，哪些能力必须受保护
- [ ] 确认是否需要区分普通登录用户和管理维护人员

### 2.3 MVP 成功标准

- [ ] 明确第一版验收不是看 Agent 数量，而是看分析结果是否可用
- [ ] 明确试点新闻能稳定命中正确行业
- [ ] 明确能召回一批基本合理的候选股票
- [ ] 明确能输出至少一条可解释推理路径
- [ ] 明确前端页面能稳定展示结构化结果和 AI 结论

## 3. P0：知识图谱前置确认

### 3.1 实体与关系冻结

- [ ] 冻结实体类型：`theme` / `industry` / `chain_node` / `company`
- [ ] 冻结关系类型：`relates_to` / `contains` / `upstream_of` / `belongs_to` / `participates_in`
- [ ] 确认 `upstream_of` 只维护一个方向，不落 `downstream_of`
- [ ] 确认 `events` 与 `event_entity_map` 作为独立表，不混入 `kg_entities`
- [ ] 确认 `securities` 作为公司到股票代码的独立映射层

### 3.2 试点行业范围

- [ ] 冻结第一阶段试点行业为 `储能 / 光伏 / 算力`
- [ ] 确认是否允许后续增加第 4 个试点行业，但不影响第一版范围
- [ ] 确认每个试点行业至少需要覆盖哪些链条节点
- [ ] 确认每个试点行业至少需要覆盖哪些核心公司
- [ ] 确认样例数据只追求“高质量结构模板”，不追求全量事实

### 3.3 Seed 录入规范

- [ ] 确认 `entity_type + canonical_name` 作为去重依据
- [ ] 确认 `canonicalRef` 作为跨 bundle 的标准引用键
- [ ] 确认别名录入规范：简称 / 英文名 / 俗称 / 同义词
- [ ] 确认关系录入规范：行业包含链条、链条上下游、公司参与环节、公司归属行业
- [ ] 确认跨行业公司可多对多映射，不强制单挂一个节点

## 4. P0：前端需要先确认的内容

### 4.1 页面展示字段

- [ ] 明确新闻详情页智能分析模块展示哪些字段
- [ ] 明确是否展示 `matchedThemes`
- [ ] 明确是否展示 `matchedIndustries`
- [ ] 明确是否展示 `matchedChainNodes`
- [ ] 明确是否展示 `candidateStocks`
- [ ] 明确是否展示 `reasoningPaths`
- [ ] 明确是否展示 `report.summary`
- [ ] 明确是否展示 `report.risks`

### 4.2 页面职责拆分

- [ ] 明确新闻详情页和智能分析页分别承担什么功能
- [ ] 明确聊天页是普通聊天，还是“带图谱增强的投研问答”
- [ ] 明确图谱页第一版是否真的要上线
- [ ] 明确历史研究记录是否在第一版展示
- [ ] 明确收藏和研究记录是否需要与用户体系绑定

### 4.3 前端状态设计

- [ ] 明确至少包含 `idle / loading / success / error`
- [ ] 明确未登录状态如何表现
- [ ] 明确分析失败时是否允许重试
- [ ] 明确结构化结果和 AI 文本结果是否分开渲染
- [ ] 明确后续是否需要支持流式返回

## 6. P0：AI 工作流前置确认

### 6.1 第一版工作流

- [ ] 冻结第一版主流程：`鉴权 -> 读输入 -> 实体识别 -> 图谱查询 -> 候选股票 -> 排序 -> 结论生成`
- [ ] 确认第一版是否先弱化复杂事件抽取
- [ ] 确认第一版是否先用规则排序，不上模型重排
- [ ] 确认第一版是否先不做人工审核节点
- [ ] 确认第一版是否先不做复杂多分支回退

### 6.2 LangGraph / LangChain 定位

- [ ] 确认 `LangGraph` 负责工作流编排，不负责主数据存储
- [ ] 确认 `LangChain` 主要用于模型、prompt、tool 封装
- [ ] 确认业务规则仍放在 `service / repository`
- [ ] 确认数据库中的事实不放进 prompt 里“硬猜”
- [ ] 确认最终输出要尽量可解释、可追溯

### 6.3 Tool 设计边界

- [ ] 确认哪些能力应该做成工具：新闻读取、图谱查询、证券映射、排序
- [ ] 确认哪些能力应该用普通代码：鉴权、规则打分、参数校验
- [ ] 确认哪些能力适合交给模型：事件总结、报告生成、自然语言解释
- [ ] 确认工具输出尽量结构化
- [ ] 确认工作流状态对象需要哪些字段

## 8. 推荐推进顺序

建议按下面顺序推进，而不是并行乱开工：

1. 产品范围与权限边界确认
2. 实体、关系、试点行业冻结
3. 页面展示字段冻结
4. API 返回结构冻结
5. 第一版工作流冻结
6. 样本新闻与验收口径准备
7. 再进入 Schema、seed、service、API 开发

---

## 9. 对你最重要的结论

你现在最该做的前置工作，不是先写 LangChain 代码，而是先把下面这些事情定清楚：

- 做哪些功能
- 给谁用
- 页面展示什么
- API 返回什么
- 图谱里存什么
- AI 工作流第一版怎么跑

只要这几件事先冻结，后面无论你是前端、后端，还是接 LangGraph，开发都会顺很多。
