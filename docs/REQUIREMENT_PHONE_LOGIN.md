# 手机号登录需求文档

## 一、需求概述

### 1.1 背景

当前 FinAgents 应用支持以下登录方式：

- **Magic Link（邮箱验证链接）**
- **GitHub OAuth**
- **Google OAuth**

为满足国内用户习惯、降低登录门槛、提升移动端体验，拟新增**手机号 + 短信验证码**登录方式。

### 1.2 目标

| 目标           | 说明                                               |
| -------------- | -------------------------------------------------- |
| 降低登录门槛   | 用户无需邮箱或第三方账号，手机号即可完成注册/登录  |
| 符合国内习惯   | 手机号登录是国内主流应用的标配能力                 |
| 与现有体系兼容 | 复用 Supabase Auth 架构，与邮箱/OAuth 用户统一管理 |
| 安全可控       | 短信验证码一次性、有时效，避免密码泄露风险         |

### 1.3 目标用户

- 无邮箱或不愿使用邮箱登录的用户
- 偏好手机号作为主登录方式的国内用户
- 移动端用户（手机号输入更便捷）

## 二、功能需求

### 2.1 核心流程

```
┌─────────────────────────────────────────────────────────────────┐
│                    手机号登录主流程                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  1. 用户输入手机号（含国家/地区码，如 +86）                        │
│  2. 点击「获取验证码」                                            │
│  3. 后端调用 Supabase signInWithOtp({ phone }) 发送短信           │
│  4. 用户收到 6 位数字验证码（有效期约 60 秒）                      │
│  5. 用户输入验证码，点击「登录」                                  │
│  6. 后端调用 verifyOtp({ phone, token, type: 'sms' }) 校验        │
│  7. 校验成功 → 建立 Session → 重定向到首页或 next 参数指定页面     │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 2.2 功能点清单

| #   | 功能点           | 优先级 | 说明                                                      |
| --- | ---------------- | ------ | --------------------------------------------------------- |
| F1  | 手机号输入       | P0     | 支持国际格式（如 +86 13800138000），带国家/地区选择器     |
| F2  | 发送验证码       | P0     | 调用 Supabase signInWithOtp 发送短信，展示倒计时（60 秒） |
| F3  | 验证码输入       | P0     | 6 位数字输入框，支持粘贴                                  |
| F4  | 验证并登录       | P0     | 调用 verifyOtp 校验，成功则建立 Session 并跳转            |
| F5  | 错误提示         | P0     | 手机号格式错误、验证码错误、发送失败等明确提示            |
| F6  | 防刷与限流       | P1     | 同一手机号 60 秒内仅可发送一次；可配置每日/每 IP 上限     |
| F7  | 账号绑定（可选） | P2     | 已登录用户将手机号与当前账号绑定，用于多端统一            |

### 2.3 业务规则

| 规则              | 说明                                                                 |
| ----------------- | -------------------------------------------------------------------- |
| 手机号格式        | 需包含国家码（如 +86），Supabase 要求 E.164 格式                     |
| 验证码有效期      | 默认 60 秒，可配置（Supabase 侧）                                    |
| 新用户自动注册    | 首次使用某手机号登录时，自动创建 auth.users 及 profiles 记录         |
| 与邮箱/OAuth 共存 | 同一用户可同时拥有邮箱、手机号、OAuth 多种登录方式（需账号绑定功能） |

## 三、非功能需求

### 3.1 性能

| 指标       | 要求                                      |
| ---------- | ----------------------------------------- |
| 验证码发送 | 用户点击后 3 秒内收到短信（取决于运营商） |
| 登录响应   | 验证码校验成功后 1 秒内完成跳转           |

### 3.2 安全

| 措施                    | 说明                                   |
| ----------------------- | -------------------------------------- |
| 验证码一次性            | 使用后立即失效                         |
| 限流                    | 防止恶意刷短信，控制成本               |
| 手机号脱敏展示          | 个人中心等场景展示为 138\*\*\*\*8000   |
| 与现有 Session 机制一致 | 复用 HttpOnly Cookie、PKCE、Token 刷新 |

### 3.3 兼容性

- 与现有 Magic Link、GitHub、Google 登录并列展示
- 登录页布局需适配移动端（输入框、按钮尺寸）
- 支持多语言（中/英）文案

```
│  │               │     │                 │     │  不走 callback) │  │
│  └──────────────┘     └────────┬────────┘     └────────────────┘  │
│                                │                                    │
└────────────────────────────────┼────────────────────────────────────┘

│                    Supabase Auth                                   │
│                                                                    │
│  ┌─────────────────┐     ┌─────────────────┐                       │

│  │ auth.users      │  phone 字段存储手机号                          │

│  └─────────────────┘                                               │
└──────────────────────────────────────────────────────────────────┘
```

### 4.2 关键 API

**发送验证码：**

```typescript
const {error} = await supabase.auth.signInWithOtp({
    phone: '+8613800138000',
    options: {shouldCreateUser: true},
});
```

**校验验证码并登录：**

```typescript
const {data, error} = await supabase.auth.verifyOtp({
    phone: '+8613800138000',
    token: '123456',
    type: 'sms',
});
// 成功后 data.session 已建立，cookie 由 @supabase/ssr 自动设置
```

> 注意：手机号 OTP 的 `verifyOtp` 在服务端调用后，Session 会写入 cookie，**无需**经过 `/auth/callback`。与 Magic Link、OAuth 不同，手机号登录是「同页完成」的。

### 4.3 文件变更清单

| 类型 | 文件                   | 改动                                                         |
| ---- | ---------------------- | ------------------------------------------------------------ |
| 新建 | `actions/auth.ts` 扩展 | 新增 `sendPhoneOtp(phone)`, `verifyPhoneOtp(phone, token)`   |
| 修改 | `login-form.tsx`       | 新增手机号 Tab/区块，含手机号输入、验证码输入、发送/登录按钮 |
| 修改 | `AUTH_LOGIN.md`        | 补充手机号登录配置说明                                       |
| 修改 | Supabase Dashboard     | 启用 Phone Provider，配置 SMS Provider                       |

### 4.4 数据库与 Profile

- `auth.users` 表由 Supabase 管理，支持 `phone` 字段
- 现有 `handle_new_user` 触发器需兼容 `phone` 用户：`name` 可取自 `phone` 脱敏或默认「手机用户」
- `profiles` 表可新增 `phone` 列（可选），用于展示和账号绑定

## 六、外部依赖

### 6.1 Supabase 配置

1. **启用 Phone Provider**
    - Dashboard → Authentication → Providers → Phone → Enable

2. **配置 SMS Provider**（任选其一）
    - Twilio
    - MessageBird
    - Vonage (Nexmo)
    - TextLocal

3. **国内手机号说明**
    - 部分 SMS 服务商对国内号码有限制或需额外资质
    - 可选方案：使用支持中国区短信的 Twilio 中国、阿里云短信 + 自定义 Auth Hook 等

### 6.2 成本估算

| 项目     | 说明                                                         |
| -------- | ------------------------------------------------------------ |
| 短信费用 | 按条计费，约 $0.01–0.05/条（因服务商和地区而异）             |
| 限流策略 | 建议每手机号每日 5–10 次，每 IP 每小时 20 次，控制成本与滥用 |

## 七、风险与限制

| 风险                 | 影响                 | 缓解措施                                   |
| -------------------- | -------------------- | ------------------------------------------ |
| 短信到达率/延迟      | 用户收不到或延迟收到 | 选择稳定服务商；提供「未收到？重发」入口   |
| 国内短信政策         | 需实名、模板审核     | 提前申请短信签名与模板，选择合规服务商     |
| 恶意刷短信           | 成本激增             | 限流、图形验证码（可选）、设备指纹（可选） |
| 手机号与邮箱用户重复 | 同一人多账号         | 后续通过账号绑定合并                       |
