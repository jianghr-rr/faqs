## 1. 文档目的

这份文档不再讨论“可选方案”，而是给出第一阶段可以直接开工的冻结结论。

目标是统一下面几件事：

- 第一版到底做什么
- 第一版不做什么
- 前端页面先做到哪一步
- API 返回哪些固定字段
- 图谱范围和 AI 工作流先做到什么程度
- 什么算 MVP 完成

这份文档的定位是：

**第一阶段开发基线。**

如果后续要改方向，优先更新这份文档，而不是让实现自己漂移。

---

## 2. MVP 一句话定义

FinAgents AI 投研系统 MVP 的定义如下：

```text
对登录用户开放
基于新闻与行业知识图谱
完成“行业命中 -> 产业链扩展 -> 候选股票召回 -> 可解释 AI 结论”
的第一版闭环能力
```

---

## 3. MVP 范围冻结

### 3.1 第一版必须包含

第一版必须包含以下能力：

1. 新闻详情页智能分析
2. 智能分析页单次分析
3. AI 投研问答基础版
4. 行业知识图谱基础查询能力
5. 登录态保护

### 3.2 第一版明确不做

第一版不做以下内容：

1. 不做复杂图谱可视化
2. 不做全市场、全行业覆盖
3. 不做自动写回主图谱的全自动更新
4. 不做复杂多 Agent 自主决策
5. 不做策略回测、组合管理、交易执行
6. 不做独立图数据库
7. 不做模型重排优先架构
8. 不做管理后台

### 3.3 第一版产品边界

第一版的产品边界是：

- 先做“分析辅助”
- 不做“自动投资决策”
- 先做“可解释推理结果”
- 不承诺“收益预测”

---

## 4. 用户与权限冻结

### 4.1 用户范围

MVP 只服务于已登录用户。

### 4.2 权限结论

以下能力仅登录后可用：

1. 新闻智能分析
2. AI 投研问答
3. 图谱深度查询
4. 个性化研究记录
5. 收藏分析结果

以下能力可继续公开：

1. 新闻浏览
2. FAQ 浏览
3. 登录页与基础产品介绍

### 4.3 接口权限

所有投研相关接口默认都应做登录校验。

未登录访问时统一返回：

```json
{
    "error": {
        "code": "UNAUTHORIZED",
        "message": "Login required"
    }
}
```

### 4.4 页面权限

页面层采用两种策略：

1. 受保护页面直接要求登录
2. 非受保护页面中的高级分析按钮，对未登录用户显示登录引导

## 5. 试点范围冻结

### 5.1 试点行业

第一阶段只做 3 个试点行业：

1. `储能`
2. `光伏`
3. `算力`

### 5.2 试点目标

试点目标不是覆盖更多行业，而是验证下面这条链路是否成立：

```text
新闻 / 用户问题
-> 命中行业
-> 命中链条环节
-> 命中核心公司
-> 映射股票代码
-> 输出可解释分析
```

### 5.3 数据原则

试点数据原则如下：

1. 高质量优先，数量第二
2. 先保证结构对，再逐步扩容
3. 允许多对多关系，不强制单路径
4. 允许同一公司跨行业参与多个环节

## 6. 知识图谱范围冻结

### 6.1 实体类型

MVP 冻结以下实体类型：

- `theme`
- `industry`
- `chain_node`
- `company`

### 6.2 关系类型

MVP 冻结以下关系类型：

- `relates_to`
- `contains`
- `upstream_of`
- `belongs_to`
- `participates_in`

### 6.3 关系维护规则

冻结以下规则：

1. `upstream_of` 只维护一个方向
2. 不单独写 `downstream_of`
3. 反向关系统一通过查询层推导
4. 公司到证券的映射不走 `kg_relations`，走 `securities`
5. 事件到实体的映射不走 `kg_relations`，走 `event_entity_map`

### 6.4 主表范围

MVP 涉及的核心表如下：

- `news`
- `kg_entities`
- `kg_entity_aliases`
- `kg_relations`
- `securities`
- `events`
- `event_entity_map`

## 7. 前端页面范围冻结

### 7.1 新闻详情页

新闻详情页为第一优先级页面。

必须展示的模块：

1. 命中行业
2. 命中产业链环节
3. 候选股票列表
4. 推理路径
5. AI 摘要
6. 风险提示

第一版不做：

1. 图谱关系大图
2. 多维交互式可视化

### 7.2 智能分析页

智能分析页为第二优先级页面。

第一版定位：

- 单次分析工作台
- 支持新闻分析和自由输入分析

必须展示的模块：

1. 输入区
2. 结构化结果区
3. AI 结论区
4. 错误提示与重试

### 7.3 AI 聊天页

聊天页为第三优先级页面。

第一版定位：

- 带图谱增强的投研问答页

必须展示的内容：

1. 文本回答
2. 引用实体
3. 推理路径
4. 候选股票

第一版不要求：

1. 长期 memory
2. 复杂会话管理
3. 多用户协作

### 7.4 图谱页

图谱页不作为第一版必须上线页面。

第一阶段只需提供图谱查询能力和基础接口，是否独立出页面可以后置。

## 8. 页面展示字段冻结

第一版前端应围绕下面这些稳定字段设计。

### 8.1 核心结构化字段

- `matchedEntities`
- `reasoningPaths`
- `candidateStocks`
- `report`

### 8.2 `matchedEntities`

建议结构如下：

```json
{
    "themes": [],
    "industries": [],
    "chainNodes": [],
    "companies": []
}
```

### 8.3 `reasoningPaths`

建议结构如下：

```json
[
    {
        "path": ["政策支持", "储能", "PCS", "阳光电源"],
        "description": "政策利好储能建设，PCS 环节弹性更强"
    }
]
```

### 8.4 `candidateStocks`

建议结构如下：

```json
[
    {
        "companyEntityId": "c1",
        "companyName": "阳光电源",
        "stockCode": "300274",
        "exchange": "SZSE",
        "score": 0.93,
        "reason": "PCS 核心参与者"
    }
]
```

### 8.5 `report`

建议结构如下：

```json
{
    "summary": "本次政策更可能利好储能产业链中的 PCS 与系统集成环节。",
    "reasoning": ["政策事件直接作用于储能建设", "储能行业中的 PCS 与系统集成环节受益更明显"],
    "risks": ["政策执行节奏仍需观察"]
}
```

## 9. API 范围冻结

### 9.1 第一版必须提供

第一版必须提供以下接口能力：

1. `POST /api/research/analyze-news`
2. `POST /api/research/analyze-query`
3. `POST /api/research/chat`

### 9.2 第一版建议提供

第一版建议提供以下图谱查询接口：

1. `GET /api/kg/search`
2. `GET /api/kg/industries/:id`
3. `GET /api/kg/companies/:id`

### 9.3 第一版不要求

第一版不要求：

1. 历史会话完整管理接口
2. 流式输出接口
3. 批量分析接口
4. 后台维护接口

## 10. AI 工作流冻结

### 10.1 第一版主流程

第一版冻结为以下工作流：

```text
鉴权
-> 读取输入
-> 实体识别
-> 图谱查询
-> 候选股票扩展
-> 规则排序
-> 生成 AI 结论
-> 返回结果
```

### 10.2 第一版简化策略

第一版采用以下简化策略：

1. 事件抽取可做轻量版本，不要求非常复杂
2. 股票排序先用规则打分
3. 不引入复杂多分支工作流
4. 不引入人工审核节点
5. 不做模型驱动的自动图谱写入

### 10.3 LangGraph 定位

MVP 中 `LangGraph` 的定位是：

- 负责工作流编排
- 负责节点状态流转
- 不负责主数据存储
- 不替代业务 service

### 10.4 LangChain 定位

MVP 中 `LangChain` 的定位是：

- 封装模型调用
- 封装 prompt
- 封装 tools

不应让 LangChain 承担：

1. 数据库事实判断
2. 登录鉴权
3. 业务主规则定义

模块职责冻结如下：

## 12. 第一版验收标准冻结

MVP 完成不看“用了多少 Agent”，而看下面这些结果。

### 12.1 技术验收

1. 已登录用户可正常访问 AI 投研相关接口
2. 未登录用户访问投研接口返回 `401`
3. 系统能基于新闻返回行业、环节、候选股票和 AI 结论
4. 系统能基于用户问题返回图谱增强后的回答
5. 前端能稳定渲染结构化结果与 AI 文本

### 12.2 业务验收

1. 对 `储能 / 光伏 / 算力` 试点新闻，行业命中基本合理
2. 对试点新闻，候选股票结果基本合理
3. 每次分析至少能输出一条可解释推理路径
4. AI 结论与结构化结果不明显冲突

### 12.3 产品验收

1. 新闻详情页分析流程可用
2. 智能分析页可用
3. 聊天页可输出带路径和股票引用的回答
4. 未登录时的交互表现明确且一致

## 13. 第一版开发顺序冻结

按下面顺序开发，不建议打乱：

1. 图谱 schema 与 seed
2. 图谱基础查询 service
3. 登录校验与 API 保护
4. `analyze-news`
5. `analyze-query`
6. `chat`
7. 新闻详情页智能分析模块
8. 智能分析页
9. 聊天页增强

---

## 14. 最终冻结结论

FinAgents AI 投研系统 MVP，第一阶段不追求“大而全”，而是冻结为一条清晰的最小闭环：

```text
登录用户
-> 输入新闻或问题
-> 系统命中行业与链条
-> 扩展候选公司和股票
-> 输出推理路径
-> 生成 AI 投研结论
```

第一阶段的目标不是做成“AI 版券商终端”，而是先做成：

**一个基于知识图谱、对登录用户开放、可解释、可继续扩展的 AI 投研 MVP。**
